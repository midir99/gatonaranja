A common user-scoped approach is to install binaries into:

~/.local/bin
That is the most standard choice on Linux. Many tools and distros already expect it, and users often add it to PATH.

Other possibilities exist, but ~/.local/bin is the usual default, much more than ~/.programs/.

Typical user-scoped layout:

binary: ~/.local/bin/gatonaranja
config/data, if needed later:
~/.config/gatonaranja/
~/.local/share/gatonaranja/
~/.cache/gatonaranja/
If you also want a user-scoped systemd service, that is usually:

service file: ~/.config/systemd/user/gatonaranja.service
Then the user manages it with:

systemctl --user daemon-reload
systemctl --user enable --now gatonaranja
systemctl --user status gatonaranja
So the common pattern is:

install binary to ~/.local/bin
install user service to ~/.config/systemd/user
Not usually:

~/.programs/
That can work, but it is not the conventional Linux location.

So my recommendation is:

binary: ~/.local/bin
user service: ~/.config/systemd/user
If you want, I can help you design the exact install layout and service file for a user-scoped install.


~/.config/systemd/user is the directory where per-user systemd service unit files are stored.

Meaning:

~ = the current user’s home directory
.config/systemd/user = that user’s own systemd unit directory
So if your username is alice, this would be:

/home/alice/.config/systemd/user
What goes there:

user-level service files like:
gatonaranja.service
Example:

~/.config/systemd/user/gatonaranja.service
How it differs from system-wide services:

user service:

~/.config/systemd/user/
managed with:

systemctl --user ...
system-wide service:

/etc/systemd/system/
managed with:

sudo systemctl ...
Why this is useful:

no root required
each user can run their own service
good fit for personal tools and bots
So in your case, if you want a user-scoped install:

binary in ~/.local/bin
service unit in ~/.config/systemd/user/gatonaranja.service
That is the standard per-user systemd setup.

es, that’s a very common first impression. systemd is often talked about as a system-wide service manager, but it also supports per-user service managers.

So there are really two common modes:

system-wide systemd

manages system services
uses sudo systemctl ...
unit files usually live in places like /etc/systemd/system
user systemd

manages services for one user only
uses systemctl --user ...
unit files usually live in ~/.config/systemd/user
That user mode is great for things like:

personal bots
background sync tools
local daemons
desktop/session services
So your idea of making gatonaranja installable without system-wide privileges fits very naturally with:

~/.local/bin/gatonaranja
~/.config/systemd/user/gatonaranja.service
One important nuance:
user services normally run in the user’s session context, so behavior can depend on whether the user session is active. If you want the user service to keep running even after logout, there is a related concept called linger, but you do not need to worry about that immediately unless it matters for your deployment story.

So yes:

systemd can be system-wide
but it can also be user-scoped
and for your bot, user-scoped systemd is actually a very reasonable fit

For a user-scoped systemd service, the usual approach is:

store the config in a user-owned environment file
point the service unit to it with EnvironmentFile=
Example config file:

~/.config/gatonaranja/gatonaranja.env
Contents:

AUTHORIZED_USERS=123456789,987654321
TELEGRAM_BOT_TOKEN=your-token-here
Example user service:

~/.config/systemd/user/gatonaranja.service
[Unit]
Description=gatonaranja Telegram bot

[Service]
Type=simple
EnvironmentFile=%h/.config/gatonaranja/gatonaranja.env
ExecStart=%h/.local/bin/gatonaranja
Restart=on-failure
RestartSec=5

[Install]
WantedBy=default.target
How the binary reads it:

systemd loads the variables from EnvironmentFile=...
those variables become normal environment variables for the process
your Go code can read them with os.Getenv(...), which you already support
Security:

yes, TELEGRAM_BOT_TOKEN is sensitive
protect the env file with filesystem permissions
Typical setup:

mkdir -p ~/.config/gatonaranja
chmod 700 ~/.config/gatonaranja
touch ~/.config/gatonaranja/gatonaranja.env
chmod 600 ~/.config/gatonaranja/gatonaranja.env
That means:

only the user can read/write the file
other users on the machine cannot read the token from the file
A few security notes:

the token should not be committed to git
the env file should not be world-readable
avoid putting the token directly in the service file, because unit files are easier to inspect and may be less private than a 600 env file
if you log config errors, never log the token value
So the standard answer is:

config file: ~/.config/gatonaranja/gatonaranja.env
service file uses EnvironmentFile=...
binary reads env vars normally
protect the env file with chmod 600
That is the usual and practical way to do this securely for a user-scoped service.
